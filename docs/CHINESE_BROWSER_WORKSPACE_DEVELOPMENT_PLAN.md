# Veilium 中文浏览器工作台完整开发方案

状态：**规划提案（Planning Proposal）**  
创建日期：2026-07-23  
目标产品：Veilium Browser Desktop  
目标界面语言：简体中文（`zh-CN`）  
建议实施阶段：Phase 5 完成并关闭后，作为 Phase 6 规划候选的首个产品化里程碑  
当前实现权限：**本文件本身不授权产品代码实施**

---

## 1. 文档目的

本方案用于把 Veilium 从当前偏工程控制台的桌面工作区，升级为普通用户能够理解、创建、启动和维护浏览器环境的中文多环境浏览器管理器。

本方案解决两个核心问题：

1. 当前桌面界面重点暴露生命周期、内核、运行时、适配器、凭据和 Evidence 等工程概念，用户打开应用后难以快速完成“创建环境并打开浏览器”的主要任务；
2. 当前用户可见文案以英文硬编码为主，没有统一的本地化层，中文化不能通过零散字符串替换可靠完成。

本方案定义：

- 产品定位和目标用户；
- 信息架构和页面结构；
- 中文化技术架构；
- 视觉与交互规范；
- 前后端改造边界；
- 分阶段实施顺序；
- 文件级修改范围；
- 测试、验收、风险和交付规则。

## 2. 治理边界与实施前提

### 2.1 当前仓库状态

编写本文件时：

- Phase 5 仍为 Active；
- PR #59 和分支 `agent/handoff-m5-3` 是当前唯一允许继续开发和文档更新的路径；
- PR #59 仍需要完整编译、测试、Wails 启动、Windows 构建和人工冒烟验证；
- `docs/ROADMAP.md` 和 `docs/PHASE_05.md` 明确把无关的大范围 UI 重设计列为 Phase 5 非目标。

因此，本文件可以在 PR #59 中作为未来产品化工作的规划材料存在，但不得被解释为在 PR #59 中立即加入大范围界面重构的授权。

### 2.2 推荐的授权方式

推荐顺序：

1. 完成 PR #59 当前已实现功能的可执行验证；
2. 修复验证发现的问题并合并 PR #59；
3. 完成 Phase 5 的正式关闭和源文档同步；
4. 以本文件为基础创建 Phase 6 规划包；
5. 在 `PRODUCT.md`、`ROADMAP.md`、新的 Phase 文档和 `STATUS.md` 中正式授权中文产品化里程碑；
6. 再开始产品代码实施。

若所有者决定把本方案提前纳入 PR #59，必须先同步修改上位源文档和 PR 范围，并重新评估 PR 体量、验证成本和回归风险。默认不推荐这样做。

### 2.3 不变的安全与产品边界

界面重构不得改变以下既有边界：

- 浏览器只从已注册并满足当前策略的内核启动；
- Provider 信任、Kernel 完整性、兼容性、Profile 健康和 Evidence 保持为不同状态；
- 不支持、冲突、缺失或无法验证的组合继续 fail closed；
- 密码和其他秘密继续只保存在操作系统凭据存储中；
- 管理界面不得显示、记录或导出明文秘密；
- 本地 API 和运行时控制继续限制在 loopback；
- 浏览器数据继续作为不透明数据处理；
- UI 不能因为隐藏了技术细节而隐藏真实风险或把有限状态显示成可用状态；
- 中文管理界面语言不得自动修改某个 Profile 的浏览器语言、时区、Accept-Language 或身份配置。

## 3. 产品重新定位

### 3.1 产品定义

Veilium 应呈现为：

> 一款本地优先、中文友好的多环境隐私浏览器管理器。用户可以创建相互隔离的浏览器环境，为每个环境配置独立的浏览器、网络和身份参数，并在明确的安全边界内启动、检测、备份和管理这些环境。

不再以“内核、运行时、适配器和生命周期控制台”作为第一视觉印象。

### 3.2 产品形态

Veilium 主程序继续作为浏览器环境管理器；真正的网页浏览仍由独立启动的 Chromium 窗口承担。

本方案不要求：

- 在 Wails 主窗口中嵌入网页内容；
- 在主窗口中实现 Chrome 式标签栏；
- 在主窗口中实现真实地址栏、下载栏或扩展栏；
- 把 Veilium 变成 Chromium 外壳。

“更像浏览器操作界面”的含义是：

- 首页围绕浏览器环境和“打开浏览器”组织；
- 用户不必理解 Provider、CDP、Evidence、operation journal 等术语才能完成日常任务；
- 高级技术能力仍可查看，但通过渐进式披露放在详情、检测报告或高级设置中。

### 3.3 目标用户

主要用户：

- 在一台设备上管理多个独立浏览器环境的个人用户；
- 需要稳定环境、代理和账号隔离的运营人员；
- 需要复现浏览器、网络和身份条件的开发与 QA 人员。

次要用户：

- 需要查看 Provider、Kernel、网络证据、运行日志和兼容性详情的高级用户；
- 后续受控自动化能力的开发者。

### 3.4 用户最重要的任务

按优先级排序：

1. 创建第一个浏览器环境；
2. 配置或选择浏览器内核；
3. 配置直连或代理；
4. 检查环境是否可启动；
5. 打开浏览器；
6. 查看运行状态并关闭浏览器；
7. 编辑、复制、归档、备份或恢复环境；
8. 对多个环境执行受限的批量操作；
9. 在出现问题时获得可理解的中文原因和修复入口；
10. 在需要时查看完整技术详情。

## 4. 成功标准

### 4.1 用户体验成功标准

- 应用启动后默认进入“浏览器环境”页；
- 首屏必须出现“新建环境”和环境列表或明确的空状态引导；
- 用户不进入高级设置即可完成创建、代理配置、健康检查和打开浏览器；
- 可启动环境的主要操作始终显示为文字按钮“打开浏览器”；
- 运行中的环境主要操作显示为“关闭浏览器”；
- 日常页面不直接使用 Provider Contract、CDP、Evidence、operation journal、fail-closed 等英文工程术语；
- 所有用户可见的固定文案默认使用简体中文；
- 关键错误必须包含“发生了什么、为什么、下一步做什么”；
- 1366×768 和 1920×1080 下核心操作不被遮挡或溢出；
- 仅使用键盘也可以完成主要操作。

### 4.2 技术成功标准

- Profile、Kernel、Adapter、Credential、Lifecycle 和 Runtime 的现有持久化格式不因界面重构而改变；
- 启动、停止、诊断、备份、恢复和批量服务仍调用现有后端权威接口；
- 本地化层拥有类型检查和缺失键测试；
- 不再在业务组件内新增直接面向用户的硬编码英文字符串；
- 后端错误逐步使用稳定错误码和结构化参数；
- 原始技术错误可在“技术详情”中查看，但不作为默认主文案；
- 现有安全、完整性、生命周期和 Evidence 回归测试继续通过。

### 4.3 产品可用性指标建议

这些指标用于人工验收或后续埋点设计，不授权加入遥测：

- 新用户从启动应用到创建环境不超过 3 个主要步骤；
- 已配置环境从列表到打开浏览器只需要 1 次主要点击；
- 代理失败时，用户可以从错误卡片直接进入“检测代理”；
- 内核缺失时，用户可以从环境行直接进入内核安装引导；
- 首次使用时不要求用户先浏览内核注册、运行会话或凭据保险库页面。

## 6. 新信息架构

### 6.1 一级导航

建议一级导航固定为五项：

1. **浏览器环境**
2. **代理与网络**
3. **数据与恢复**
4. **批量管理**
5. **设置**

默认入口：**浏览器环境**。

### 6.2 旧入口到新入口映射

| 当前入口 | 新位置 | 用户可见名称 |
| --- | --- | --- |
| Overview | 合并到浏览器环境页顶部 | 环境概览 |
| Browser profiles | 一级入口 | 浏览器环境 |
| Local recovery | 一级入口 | 数据与恢复 |
| Runtime sessions | 浏览器环境页的运行筛选、会话抽屉及设置高级页 | 运行会话 |
| Kernel registry | 设置 → 高级设置 | 浏览器内核 |
| Proxy adapters | 代理与网络 → 高级设置 | 代理组件 |
| Credential vault | 设置 → 安全与凭据 | 凭据管理 |
| Settings | 一级入口 | 设置 |
| Multi-Profile tools | 一级入口，取消悬浮按钮 | 批量管理 |

### 6.3 导航原则

- 一级导航按用户任务组织，不按后端模块组织；
- 技术名词不在一级导航出现；
- 同一能力只保留一个主要入口；
- 详情页可提供上下文跳转，例如“缺少内核 → 前往安装”；
- 不使用右下角常驻的大型浮动按钮遮挡内容；
- 桌面运行状态保留在侧边栏底部，但改为中文且可点击查看详情。

## 7. 页面与交互规格

## 7.1 应用外壳

### 页面框架

- 左侧固定导航；
- 顶部显示当前页面标题、全局搜索入口、运行状态和版本；
- 主内容区使用统一最大宽度、间距和滚动规则；
- 全局通知使用顶部消息条或右上角通知中心；
- 危险确认使用统一模态框，不直接依赖浏览器原生 `window.confirm`。

### 顶部操作

建议保留：

- 当前页面标题；
- 全局刷新；
- 当前桌面运行状态；
- 版本号；
- 帮助或“技术详情”入口。

不建议在顶部长期展示无上下文的工程状态。

### 侧边栏底部状态

中文显示示例：

- `桌面运行正常`；
- `本地进程控制已启用`；
- 浏览器预览模式下显示 `预览模式，无法启动本地浏览器`。

点击后打开运行状态抽屉，展示：

- Wails 桌面模式；
- 活动会话数；
- 最近错误；
- 本地服务状态；
- 进入高级运行详情的链接。

## 7.2 浏览器环境页

这是默认首页和产品核心页面。

### 顶部区域

标题：`浏览器环境`  
说明：`创建和管理彼此隔离的浏览器环境。每个环境拥有独立的浏览器数据、网络和身份配置。`

主要操作：

- `+ 新建环境`；
- `导入环境`；
- 可选的 `从模板创建`。

### 概览卡片

仅保留用户可理解的统计：

- 全部环境；
- 运行中；
- 可启动；
- 需要处理。

“需要处理”合并内核缺失、代理失败、生命周期限制、完整性问题和恢复要求，但点击后必须显示具体原因分类。

### 搜索与筛选

- 搜索环境名称、分组或标签；
- 分组；
- 状态；
- 代理状态；
- 浏览器版本；
- 最近使用时间；
- 重置筛选。

筛选条件必须以中文标签展示，内部枚举值保持不变。

### 列表字段

建议桌面表格字段：

- 选择框；
- 环境名称；
- 分组和标签；
- 网络；
- 浏览器；
- 最近使用；
- 状态；
- 操作。

窄窗口下改为卡片布局，优先保留名称、状态、网络和主要操作。

### 环境行状态

每行最多显示一个主状态和一个次级说明：

- `运行中`；
- `可启动`；
- `待配置`；
- `受限`；
- `需要恢复`；
- `已归档`；
- `回收站中`。

鼠标悬停或展开详情后显示内部状态：

- 生命周期状态；
- Kernel 完整性；
- Provider 信任；
- 代理检查；
- Profile 健康；
- Evidence 适用性。

### 行操作

主按钮：

- 可启动时：`打开浏览器`；
- 运行中：`关闭浏览器`；
- 待配置时：`继续配置`；
- 受限时：`查看问题`。

次级操作：

- `编辑`；
- `更多` 菜单。

更多菜单包括：

- 复制环境；
- 检测代理；
- 环境检测；
- 查看启动详情；
- 创建快照；
- 归档；
- 移入回收站。

禁止继续在一行中并排放置大量无文字小图标。

### 空状态

无环境：

> 还没有浏览器环境。创建一个独立环境，配置浏览器和网络后即可打开。

操作：

- `创建第一个环境`；
- `导入环境配置`。

搜索无结果：

> 没有符合当前筛选条件的环境。

操作：

- `清除筛选`；
- `新建环境`。

### 环境详情抽屉

点击环境名称打开抽屉，分为：

1. 概览；
2. 网络；
3. 浏览器与身份；
4. 数据与恢复；
5. 检测报告；
6. 技术详情。

技术详情中才显示 Provider、Kernel SHA-256、CDP、Evidence、operation ID 和原始错误。

## 7.3 新建与编辑环境

现有单一长表单改为分步流程，并支持保存草稿。

### 步骤 1：基本信息

字段：

- 环境名称；
- 分组；
- 标签；
- 备注。

默认值应足够让用户直接继续，不要求理解生命周期状态。

### 步骤 2：浏览器

普通模式：

- 已安装浏览器；
- 浏览器版本；
- 操作系统身份；
- 界面语言；
- 时区；
- 分辨率。

规则：

- 默认选择可用且策略允许的已验证内核；
- 如果没有可用内核，显示安装引导；
- 不在普通模式中要求用户选择 Provider；
- Provider 信任和版本合同放入“高级详情”。

### 步骤 3：网络代理

选项：

- 直连；
- HTTP；
- HTTPS；
- SOCKS5；
- 现有受支持的 Xray 或 sing-box 路由。

字段按路由类型动态显示：

- 地址；
- 端口；
- 凭据引用；
- 代理组件；
- 测试按钮。

检测结果：

- 是否可连接；
- 延迟；
- 出口 IP；
- DNS 路由；
- WebRTC 策略；
- 可理解的失败原因。

密码输入后直接交给操作系统凭据存储，不在 Profile 表单状态、日志或持久化元数据中回显。

### 步骤 4：身份设置

默认提供：`推荐配置`。

普通模式只显示：

- 平台；
- 语言；
- 时区；
- 屏幕参数；
- 与当前 Provider 合同明确支持的选项。

高级设置显示：

- 完整能力清单；
- 每项支持状态；
- 不支持原因；
- Evidence 或兼容性范围。

不允许通过 UI 暗示未验证能力已经生效。

### 步骤 5：检查并创建

显示：

- 环境名称；
- 浏览器；
- 网络；
- 身份摘要；
- 阻止创建或启动的问题；
- 仅限提示的问题。

底部操作：

- 返回；
- 保存草稿；
- 创建环境；
- `创建并打开浏览器`。

“创建并打开浏览器”必须先使用后端权威检查；如果创建成功但启动失败，应保留已创建环境并显示明确的启动失败原因。

### 编辑模式

- 运行中的环境默认禁止修改会影响启动的配置；
- 提示用户先关闭浏览器；
- 只读字段要解释原因；
- 保存前重新校验生命周期、Kernel、路由和能力合同；
- 保存失败不得丢失表单输入。

## 7.4 代理与网络页

### 用户层功能

- 已保存的代理配置；
- 当前被哪些环境引用；
- 连接检测；
- 延迟与出口 IP；
- 凭据引用状态；
- 新建、编辑、复制和停用代理配置。

### 高级功能

折叠到“代理组件”区域：

- Xray 注册和验证；
- sing-box 注册和验证；
- 固定官方版本安装；
- SHA-256、来源和许可证；
- 原生配置检查。

### 许可证交互

- 软件许可证原文、项目名称和 SPDX 标识保持原样；
- 中文解释可以补充，但不能替代法律文本；
- 用户必须明确勾选后才允许下载；
- 不进行自动下载或静默更新。

## 7.5 数据与恢复页

将现有 Local recovery 能力按用户任务重新分区：

1. **环境备份**：创建和查看本机快照；
2. **恢复环境**：从已验证快照恢复为新环境；
3. **归档环境**：可逆归档和取消归档；
4. **回收站**：恢复或执行需要精确确认的永久删除；
5. **操作记录**：进度、历史、取消和恢复要求；
6. **存储检查**：只读容量、缺失、孤立和不安全项。

关键交互：

- 所有破坏性操作使用统一中文确认对话框；
- 精确确认短语保持后端要求，但界面必须解释为什么需要输入；
- “需要恢复”状态必须在页面顶部突出显示；
- 不提供自动修复、自动删除或通用文件系统浏览器；
- 快照、回收站和原环境的关系必须可视化说明。

建议页签：

1. 批量操作；
2. 生命周期；
3. 环境属性；
4. 健康检查；
5. 导入与导出；
6. 模板；
7. 操作记录。

交互原则：

- 用户先固定选择环境，再选择操作；
- 明确显示已选择数量和上限；
- 显示不符合条件的环境及原因；
- 执行前显示预检；
- 执行中显示每个环境的结果；
- 取消只在安全边界可用；
- 不提供批量启动；
- 不提供批量永久删除；
- 批量移入回收站继续要求精确固定选择确认；
- 报告导出继续进行脱敏并禁止覆盖已有文件。

## 8. 简体中文与本地化架构

## 8.1 总体策略

第一版采用“简体中文默认、英文回退”的双语言基础架构。即使当前产品只面向中文用户，也不能继续把中文直接散落到各组件中。

推荐先使用零额外运行时依赖的类型化本地化层，利用 TypeScript 和浏览器原生 `Intl`：

```text
frontend/src/i18n/
├── index.ts
├── locale.ts
├── keys.ts
├── zh-CN.ts
├── en.ts
├── format.ts
├── status.ts
└── errors.ts
```

当后续确认需要复杂复数、远程语言包或更多语言时，再通过 ADR 评估 `i18next` 等成熟库；本里程碑不为库而库。

### 8.2 翻译函数

提供：

- `t(key, params)`：固定文案和插值；
- `formatDateTime(value)`；
- `formatRelativeTime(value)`；
- `formatBytes(value)`；
- `formatCount(value, key)`；
- `statusLabel(domain, value)`；
- `errorMessage(code, params)`。

所有翻译键必须具备类型提示。禁止以任意字符串访问字典。

### 8.3 文案覆盖范围

必须本地化：

- 导航；
- 标题和说明；
- 按钮；
- 输入框占位符；
- 表单标签；
- 下拉选项；
- 空状态；
- Tooltip；
- 状态标签；
- 确认对话框；
- 加载和进度；
- 错误与警告；
- 操作历史显示名称；
- 诊断结论；
- 可取消状态；
- 日期、时间、大小和数量；
- 无障碍标签；
- HTML 标题和启动占位文案。

### 8.4 不应翻译或需谨慎翻译的内容

保持原值：

- 用户输入的环境名称、分组、标签和备注；
- Profile ID、operation ID、SHA-256；
- Chromium、Xray、sing-box 等产品名；
- HTTP、HTTPS、SOCKS5、VMess、VLESS、Trojan、Shadowsocks 等协议名；
- SPDX 许可证标识；
- 文件路径；
- 后端内部枚举和持久化字段。

许可证法律文本不得因翻译而改变含义。可同时显示中文说明和原始许可证标识。

### 8.5 管理语言与环境语言分离

管理界面的 `zh-CN` 只影响 Veilium 自身。

以下配置继续属于每个 Profile：

- Chromium 界面语言；
- Accept-Language；
- 时区；
- 地区；
- 平台和身份配置。

切换 Veilium 管理语言不得批量修改 Profile，也不得改变已生成的启动计划。

### 8.6 HTML 与系统设置

- `frontend/index.html` 的 `lang` 改为 `zh-CN`；
- 标题改为 `Veilium 浏览器`；
- 启动占位文案改为中文；
- 如支持语言切换，运行时同步 `document.documentElement.lang`；
- 语言偏好只保存非敏感 locale 标识，可使用本地偏好存储。

### 8.7 本地化质量门禁

- `zh-CN` 和 `en` 必须具有完全相同的键集合；
- 测试失败时显示缺失键；
- 生产构建不得静默显示键名；
- 开发模式可对缺失翻译抛出明显错误；
- 新增面向用户文案必须先进入语言目录；
- 可增加静态扫描，阻止在指定业务组件中新增明显硬编码英文文案。

## 9. 中文文案规范

### 9.1 术语原则

- 优先使用用户任务语言；
- 同一概念全局只使用一个中文名称；
- 只有在高级详情中保留英文技术名；
- 错误不使用模糊的“操作失败”，必须说明对象和下一步；
- 不把“受支持”写成“保证不被识别”或其他未经证据支持的宣传。

### 9.2 建议术语表

| 内部或现有英文 | 用户界面中文 | 高级详情说明 |
| --- | --- | --- |
| Profile | 浏览器环境 | 技术详情可写 Profile |
| Browser profile | 浏览器环境 | 不使用“浏览器配置文件”作为主称呼 |
| Runtime session | 运行会话 | 一个已启动的浏览器进程会话 |
| Kernel | 浏览器内核 | 高级页保留 Kernel 标识 |
| Provider | 浏览器提供方 | 仅高级详情显示 |
| Proxy adapter | 代理组件 | Xray 或 sing-box 本地运行组件 |
| Credential vault | 凭据管理 | 操作系统凭据存储 |
| Lifecycle | 环境状态 / 生命周期 | 普通页使用“环境状态” |
| Evidence | 检测证据 / 技术证据 | 仅高级检测报告显示 |
| Launch plan | 启动详情 | 不使用“启动计划”作为主按钮 |
| Snapshot | 快照备份 | 强调本机恢复用途 |
| Recoverable trash | 回收站 | 明确可恢复 |
| Recovery required | 需要恢复 | 高优先级警告 |
| Ready | 可启动 | 不写“准备好” |
| Draft | 待配置 | 根据上下文可显示“草稿” |
| Archived | 已归档 | 不可直接启动 |
| Trashed | 回收站中 | 可恢复 |
| Invalid | 无效 | 必须附带原因 |
| Verify | 验证完整性 | 按对象调整文案 |
| Diagnostics | 检测 | 技术详情可写 Diagnostics |

### 9.3 错误文案结构

每条关键错误包含：

1. 标题：发生了什么；
2. 原因：为什么被阻止；
3. 操作：下一步去哪里；
4. 技术详情：错误码和原始信息，可折叠。

示例：

> **无法打开浏览器**  
> 当前环境选择的浏览器内核尚未通过完整性验证。  
> 请前往“设置 → 浏览器内核”完成验证后重试。  
> 技术代码：`KERNEL_NOT_VERIFIED`

## 12. 后端与桌面 API 调整

## 12.1 原则

大部分后端服务保持不变。界面重构不是重写业务逻辑的理由。

只在以下情况下调整 Go/Wails API：

- 当前错误只有英文自由文本，无法可靠映射；
- 页面需要后端已有但未投影的安全状态；
- 多个前端请求可能造成状态竞态，需要后端聚合预检；
- 用户动作必须由后端再次授权，不能只依赖前端禁用按钮。

### 12.2 结构化错误

建议统一桌面错误结构：

```json
{
  "code": "KERNEL_NOT_VERIFIED",
  "message": "internal English diagnostic",
  "params": {
    "kernelId": "..."
  },
  "retryable": false,
  "action": "open-kernel-settings"
}
```

约束：

- `code` 稳定且可测试；
- `params` 不包含密码、令牌或私有运行配置；
- `message` 用于技术详情和日志，不作为默认用户文案；
- 前端根据 `code` 和 `params` 生成中文；
- 未知错误显示安全的中文回退，并允许复制脱敏技术详情。

### 12.3 预检接口

优先复用现有 launch plan、lifecycle preflight、recovery preflight 和 health report。

若需要新增，建议是聚合视图而非新权威状态：

- 环境可启动摘要；
- 新建环境提交前预检；
- 代理配置测试摘要；
- 修复入口建议。

任何摘要都必须从现有权威记录计算，不能成为独立持久化真相。

### 12.4 枚举与显示

后端继续返回稳定英文枚举：

- `available`；
- `draft`；
- `archived`；
- `running`；
- `verified`；
- `limited` 等。

前端负责显示中文。禁止把中文枚举写入 JSON、Go 常量或持久化文件。

## 13. 数据、兼容性与迁移

### 13.1 数据迁移

本方案默认不修改：

- Profile schema；
- lifecycle records；
- operation journal；
- snapshot manifest；
- template and portable profile format；
- Kernel and Adapter records；
- credential references。

因此不应需要持久化数据迁移。

### 13.2 本地偏好

允许新增非敏感界面偏好：

- locale；
- 侧边栏折叠状态；
- 界面密度；
- 是否显示高级功能；
- 最后打开的一级页面。

偏好应：

- 有默认值；
- 解析失败时安全回退；
- 不包含 Profile 数据、路径、秘密或运行令牌；
- 可以清除而不影响业务数据。

### 13.3 兼容性

- 已有环境在新界面中应原样显示；
- 旧环境状态不应因中文化自动改变；
- 未知枚举必须显示“未知状态”并提供原始值；

## 18. 风险与缓解措施

### 风险 1：把 UI 重构塞入 PR #59

影响：PR 已有大量 Phase 5 后端和前端变更，继续扩大会增加编译、回归和审查风险。

缓解：本文件先作为规划材料；Phase 5 关闭后单独授权和实施。

### 风险 2：为了“简化”而隐藏真实安全状态

影响：用户可能把受限或未验证环境误认为可安全启动。

缓解：普通界面使用可理解的摘要，高级详情保留完整状态；所有阻止条件继续由后端决定。

### 风险 3：翻译不完整

影响：中英文混杂，错误和高级功能仍不可理解。

缓解：类型化键、键集合测试、硬编码扫描和页面级人工检查。

### 风险 4：中文文案导致布局溢出

影响：按钮、表格、模态框和侧边栏错位。

缓解：提高字号、使用弹性布局、限制路径显示、覆盖三种窗口尺寸并测试长文本。

### 风险 5：前端重复业务规则

影响：按钮状态与后端实际权限不一致。

缓解：前端只做提示和预展示；最终授权和预检仍由后端完成。

### 风险 6：结构化错误改造范围过大

影响：多个 Go 包和测试被同时修改。

缓解：先建立前端兼容适配器；按高频错误逐步引入稳定错误码，未知错误保留安全回退。

### 风险 7：管理语言影响浏览器身份

影响：切换 Veilium 语言意外改变 Profile 的语言或 Accept-Language。

缓解：界面偏好和 Profile 配置使用不同模型、存储和 API；增加回归测试。

### 风险 8：引入过多新依赖

影响：供应链、构建和维护成本上升。

缓解：默认使用 TypeScript、React 和原生 `Intl`；每个新依赖必须有明确收益和安全审查。

### 风险 9：新页面仍然以模块为中心

影响：只是把英文控制台翻译为中文控制台，仍不符合用户任务。

缓解：所有页面验收以用户流程为准；默认首页必须围绕环境和打开浏览器组织。

- 不以静态审查替代实际运行验证。

## 20. 建议的第一项正式实施任务

Phase 5 关闭后，建议第一个正式任务为：

> **U1 — 建立简体中文本地化基础和统一设计令牌，不改变现有业务行为。**

该任务边界最清晰，风险最低，并为后续页面重构提供必要基础。

验收重点：

- 中文应用外壳；
- `zh-CN` 和 `en` 类型化字典；
- HTML 和标题中文化；
- 状态和格式化基础；
- 中文字体、字号和颜色令牌；
- 缺失翻译测试；
- 不修改 Profile、Kernel、Adapter、Runtime、Lifecycle 或 Recovery 业务逻辑。

## 21. 完成定义

只有同时满足以下条件，本方案对应的产品化工作才算完成：

- 产品不再以工程控制台作为默认体验；
- 浏览器环境是明确的核心对象；
- 用户能在中文界面创建、检测、打开、关闭、备份和管理环境；
- 高级用户仍能查看完整技术状态；
- 中文化具有可维护的架构，而不是散落字符串；
- 现有安全、隐私、完整性、生命周期和 Evidence 边界没有被弱化；
- 所有声称通过的构建、测试和人工验证均有实际执行记录；
- 源文档、代码、界面和验收结果保持一致。
